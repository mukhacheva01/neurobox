# НейроБокс — предложения по улучшению функционала

Краткий аудит кода и UX (февраль 2025). Приоритет: **В — высокий**, **С — средний**, **Н — низкий**.

**Статус (реализовано):** 1–12 — внедрено.

---

## 1. Текст: поддержка всех моделей из выбора (В)

**Проблема:** В экране выбора модели есть Gemini, Claude, DeepSeek, Grok, но `bot/providers/openai_text.py` поддерживает только **OpenAI** и **Grok**. Остальные модели фактически не вызываются (в коде провайдера — ветка только для `grok-2`, иначе OpenAI). Пользователь платит за «GPT-4.1», но при выборе Gemini/Claude/DeepSeek логика может быть неочевидной.

**Что сделать:**
- Либо добавить провайдеры (или единый роутер): **Google AI (Gemini)**, **Anthropic (Claude)**, **DeepSeek API** — ключи уже есть в `config/settings.py`.
- Либо временно скрыть неподдерживаемые модели в `model_select.py` и в `balance.py` (список моделей), чтобы не вводить в заблуждение.

**Где менять:** `bot/providers/` — новый файлы или расширение `openai_text.py`; `bot/handlers/model_select.py`, `bot/services/credits.py` (CREDIT_PRICES, FREE_MODELS).

---

## 2. Текст: стриминг ответа (В, UX)

**Проблема:** Ответ приходит одним сообщением после полной генерации. Для длинных ответов это увеличивает ощущаемое время ожидания.

**Что сделать:** Использовать streaming API OpenAI (и при необходимости других провайдеров): отправлять сообщение «печатает…», затем обновлять его по мере прихода чанков (`edit_message_text` с throttle 1–2 сек или отправка финального сообщения после завершения).

**Где менять:** `bot/providers/openai_text.py` (stream=True, обработка потока); `bot/handlers/text.py` — цикл по чанкам и обновление сообщения.

---

## 3. Стилизация фото /style (С)

**Проблема:** В гайде и в подсказке по фото указано «/style описание», но в `photo_tools.py` команда — заглушка: «в разработке».

**Что сделать:** Реализовать через fal.ai image-to-image (если есть подходящий endpoint) или через текущий `/img` с промптом вида «в стиле: {описание}, исходное изображение: [фото]». Альтернатива: один фиксированный набор стилей (аниме, масло, пиксель) кнопками под фото.

**Где менять:** `bot/handlers/photo_tools.py`, при необходимости `bot/providers/falai_image.py` или отдельный провайдер для img2img.

---

## 4. Реферальные уровни (С)

**Проблема:** В `/ref` написано «Уровни: Бронза (×1.5), Серебро (×2), Золото (×2.5), Бриллиант (×3) — в разработке».

**Что сделать:** Ввести уровни по количеству приведённых друзей (например 3 / 10 / 25 / 50). При начислении CR за реферала умножать бонус реферера на коэффициент уровня. Хранить уровень в `users` или считать по `referral_count` на лету.

**Где менять:** `bot/services/credits.py` (apply_promocode / начисление за реферала), `bot/handlers/ref_promo_stats.py` — отображение текущего уровня и прогресса до следующего.

---

## 5. Админ: отчёт по оценкам ответов 👍👎 (С)

**Проблема:** В таблице `response_ratings` сохраняются оценки, но в админке нет сводки.

**Что сделать:** В админ-панели (или отдельной команде для админов) вывести за период: количество up/down, долю положительных, при желании — по моделям (если привязать rating к модели/типу ответа). Помогает оценивать качество ответов и моделей.

**Где менять:** `bot/handlers/admin/` (новый раздел или расширение stats) — запросы к `response_ratings` (и при необходимости к `chat_messages`/факту модели).

---

## 6. Документы: загрузка PDF/DOCX (С)

**Проблема:** Сейчас `/doc` работает только с текстом в сообщении (описание → генерация .txt). Нет приёма файлов.

**Что сделать:** Принимать документ (document) от пользователя; для PDF/DOCX — извлечь текст (PyMuPDF / python-docx или внешний API), отправить в LLM для суммаризации/вопросов; ответ и при желании краткий файл — как сейчас. Ограничить размер файла (например 5–10 MB) и объём текста.

**Где менять:** `bot/handlers/doc_handler.py` — хендлер на `F.document`; новый утилитный модуль извлечения текста; учёт CR за «документ».

---

## 7. Кнопка «Апскейл» под сгенерированной картинкой (Н)

**Проблема:** В гайде может упоминаться улучшение качества; в интерфейсе после генерации картинки такой кнопки нет.

**Что сделать:** Под сообщением с картинкой добавить inline-кнопку «🔍 Апскейл» (или «Улучшить качество»). По нажатию — вызов fal.ai (или другого) upscale по `image_url`, списание CR, отправка результата. Один уровень апскейла (например 2×) достаточно для старта.

**Где менять:** `bot/handlers/image.py` — клавиатура под ответом; новый вызов в `bot/providers/falai_image.py` или отдельный endpoint.

---

## 8. Лимит длины промпта (Н, надёжность)

**Проблема:** Очень длинное сообщение может приводить к лимитам API и к большому расходу токенов.

**Что сделать:** Проверять длину `prompt` (например, 8–12k символов). При превышении — ответ «Промпт слишком длинный. Сократи до N символов» и не списывать CR. Опционально — показывать текущую длину при вводе (только в веб-клиенте; в Telegram сложнее).

**Где менять:** `bot/handlers/text.py`, при необходимости общий хук в одном месте для всех текстовых промптов.

---

## 9. Команда /feedback (Н)

**Проблема:** Нет единого канала сбора предложений и багов от пользователей.

**Что сделать:** Команда `/feedback Текст` — сохранять в БД (таблица `feedback`: user_id, text, created_at) и/или пересылать в отдельный чат админам. В ответ — «Спасибо, мы учтём».

**Где менять:** Новый хендлер (например в `start.py` или отдельный `feedback.py`), миграция для таблицы, при желании — уведомление админов.

---

## 10. Кеш баланса в Redis (Н, производительность)

**Проблема:** Каждое действие (текст, картинка и т.д.) делает запрос к БД за балансом/моделью. При высокой нагрузке — лишняя нагрузка на PostgreSQL.

**Что сделать:** После `get_or_create_user` и `spend_credits` кешировать в Redis «bought» и «free» на 1–2 минуты по ключу `balance:{user_id}`. При списании/пополнении — инвалидировать кеш. Использовать кеш только для отображения; списание по-прежнему через БД с блокировкой.

**Где менять:** `bot/services/credits.py` — обёртки чтения баланса; при наличии Redis (уже используется для FSM) — простой get/set.

---

## 11. Webhook ЮKassa (Н, документация/деплой)

**Проблема:** В коде есть `process_payment_webhook(payment_id)` для серверной обработки вебхука, но бот работает на long polling. Для приёма вебхуков нужен отдельный HTTP endpoint.

**Что сделать:** Либо поднять минимальный aiohttp/FastAPI endpoint на отдельном порту только для вебхука и вызывать `process_payment_webhook`; либо явно описать в DEPLOY.md, что при текущей схеме (polling) вебхуки не вызываются и оплаты подтверждаются по таймауту/ручной проверке. Тогда не будет расхождений между кодом и реальностью.

**Где менять:** Документация деплоя; при необходимости — отдельный скрипт или маршрут в основном приложении (если перейти на webhook для бота).

---

## 12. Напоминание об окончании подписки (Н)

**Проблема:** Если в будущем появятся подписки с датой окончания, пользователю полезно напоминание за 1–3 дня.

**Что сделать:** При наличии таблицы подписок с `end_date` — фоновый цикл (аналогично напоминаниям чата): раз в день искать подписки с `end_date` через 1–3 дня и отправлять «Подписка истекает N числа. Продлить: /balance».

**Где менять:** По появлению подписок — новый сервис или расширение `subscribe_remind`; миграции для подписей.
