# Универсальный план тестирования Telegram-бота

**Объект:** Telegram-бот [универсальный: команды, меню, inline/keyboard, интеграции с API/БД/платежами].  
**Платформа:** Telegram Bot API.

## 2. Тестирование edge cases

| ID | Ситуация | Шаги | Expected | Приоритет | Возможные баги / рекомендации |
|----|----------|------|----------|-----------|-------------------------------|
| E-01 | Очень длинное сообщение | Отправить текст ~4000+ символов (или на границе лимита бота) | Отказ с указанием лимита или обрезка; бот не падает | High | Падение/таймаут → проверять длину до вызова API, лимит в конфиге |
| E-02 | Только эмодзи | Отправить сообщение из одних эмодзи (например 🔥🔥🔥) | Обработка или явная валидация («Введите текст»), не падение | Medium | Ошибка парсинга → учитывать Unicode, нормализация |
| E-03 | Спецсимволы и Unicode | Отправить строку с \n, \t, кавычками, скриптами (арабский, иероглифы) | Корректное отображение или экранирование; нет инъекций | High | XSS/инъекции → экранировать вывод в HTML/Markdown |
| E-04 | Пустой ввод в обязательном поле | В форме оставить поле пустым и отправить | Валидация «Заполните поле» | High | См. F-13 |
| E-05 | Спам (много сообщений подряд) | Отправить 20–50 сообщений за минуту | После N сообщений — «Слишком много запросов. Подождите.»; бот не падает | High | Нет лимита → rate limit по user_id (окно 60 сек) |
| E-06 | Одновременные запросы от нескольких юзеров | 5–10 пользователей выполняют одно и то же действие одновременно | Все получают ответ, нет взаимного влияния, нет падения сервиса | High | Блокировки/гонки → не держать глобальные lock’и на время внешних вызовов |
| E-07 | Один пользователь — два устройства | Один аккаунт открыт на телефоне и десктопе, отправить запрос с обоих подряд | Оба обработаны или один «уже выполняется» (если предусмотрена защита от дубля) | Medium | Двойное списание → lock по user_id (Redis) с TTL |
| E-08 | Сообщение без username | Юзер с пустым @username отправить команду | Бот отвечает, не падает при обращении к message.from_user.username | Medium | AttributeError → использовать (message.from_user.username or "").strip() |
| E-09 | Очень длинный callback_data | [Если бот генерирует длинные callback_data] Нажать кнопку с data ~64+ байт | Telegram лимит 64 байта — кнопка может не работать или обрезаться | Low | Ошибка при нажатии → хранить состояние в БД/Redis, в callback — только id |

## 3. Тестирование производительности

| ID | Сценарий | Шаги | Expected | Метрика | Приоритет | Рекомендации |
|----|----------|------|----------|---------|-----------|---------------|
| P-01 | Задержка ответа (одиночный запрос) | Отправить одну команду/действие | Ответ в течение N сек (например &lt; 5 для простых, &lt; 30 для LLM) | Время от отправки до первого байта ответа | High | Таймауты на все внешние вызовы; логировать latency |
| P-02 | Несколько сообщений подряд | Отправить 5–10 сообщений с интервалом 1–2 сек | Все обработаны, ответы приходят по порядку или с понятной очередью | Нет потери сообщений, нет 500 в логах | High | Очередь/воркеры при тяжёлой обработке |
| P-03 | Пиковая нагрузка | 20–50 юзеров за 1 мин отправляют /start и по одному действию | Все получают ответ, время ответа в пределах SLA; нет падения процесса | RPS, p95 latency, error rate | High | Горизонтальное масштабирование, лимиты на внешние API |
| P-04 | Долгая операция (например генерация) | Запустить действие с таймаутом 30+ сек | «Ожидайте» / прогресс; по истечении таймаута — сообщение об ошибке и при необходимости откат | Таймаут на стороне бота и внешнего API | High | asyncio.wait_for, отмена задачи, refund при оплате |
| P-05 | Обработка очереди (если есть воркеры) | Поставить в очередь 10 задач, дождаться завершения | Все задачи выполнены, нет дублей, порядок или приоритет соблюдён | Throughput, failed jobs | Medium | Идемпотентность задач, retry с backoff |

## 4. Тестирование безопасности

| ID | Проверка | Шаги | Expected | Приоритет | Возможные баги / рекомендации |
|----|----------|------|----------|-----------|-------------------------------|
| S-01 | SQL-инъекция | В полях ввода подставить `'; DROP TABLE users; --` или аналог | Данные не удаляются, ввод трактуется как строка или отклонён | High | Параметризованные запросы / ORM, никогда не склеивать SQL с вводом |
| S-02 | XSS (если HTML/Markdown) | Ввести `<script>alert(1)</script>` или `[link](javascript:...)` в текст, отображаемый ботом | Экранирование или санитизация, скрипт не выполняется | High | Экранировать HTML или использовать безопасное подмножество Markdown |
| S-03 | Защита от флуда | См. E-05 | Rate limit по user_id, ответ «Подождите» | High | Middleware rate limit, скользящее окно |
| S-04 | Конфиденциальные данные в логах | Выполнить действие с токеном/паролем/картой, проверить логи | Токены/пароли не выводятся (маскировка или отсутствие в логе) | High | Redact в structlog/logger по ключам token, password, secret |
| S-05 | Доступ к админ-функциям | Обычный юзер отправить /admin или callback админки | Отказ или «Нет доступа» | High | Проверка admin_id_list / ролей перед выполнением |
| S-06 | Инъекция в callback_data | [Если callback парсится как команда] Подставить `buy_credits:1;drop` | Обрабатывается только допустимая часть или отклонено | Medium | Валидация/whitelist значений callback_data, не eval |

## 5. Тестирование интеграций

| ID | Сценарий | Шаги | Expected | Приоритет | Рекомендации |
|----|----------|------|----------|-----------|---------------|
| I-01 | Внешний API недоступен | Отключить сеть до сервера API или мок 5xx | Сообщение пользователю «Сервис временно недоступен» (без деталей), при оплате — не списывать или refund | High | Таймауты, retry с backoff, circuit breaker, fallback |
| I-02 | API возвращает неверный формат | Мок возвращает пустой JSON или не тот schema | Бот не падает, ответ «Ошибка. Попробуйте позже» или аналог | High | Валидация ответа (pydantic/схема), try/except |
| I-03 | Таймаут API | Мок не отвечает 60+ сек | Таймаут на стороне бота (например 30 сек), сообщение об ошибке | High | asyncio.wait_for или timeout в httpx/aiohttp |
| I-04 | Платежи: webhook дублируется | Дважды отправить один и тот же webhook (payment_id) | Зачисление один раз (идемпотентность) | High | payment_lock по payment_id, проверка «уже обработан» |
| I-05 | БД недоступна | Остановить Postgres/Redis на время запроса | Ошибка не пробрасывается в юзера как 500; лог + повтор при следующем запросе | High | Connection pool, retry, graceful degradation |
| I-06 | 429 от API (rate limit) | Имитировать 429 от внешнего сервиса | Ретрай с backoff или сообщение «Сервис перегружен. Попробуйте позже.» | Medium | Обработка RateLimitError, не бесконечный retry |

## 8. Автоматизированное тестирование

### 8.1 Инструменты

| Инструмент | Назначение | Когда использовать |
|------------|------------|--------------------|
| **pytest + aiogram** | Unit/интеграция хендлеров (моки Bot, Update) | Проверка логики без реального Telegram |
| **Telethon / Pyrogram** | Клиент от имени пользователя (userbot) | E2E: отправка сообщений и проверка ответов бота |
| **pytest + httpx/aiohttp** | Моки внешних API и webhook | Интеграции, платежи, LLM |
| **locust / pytest-benchmark** | Нагрузка (много апдейтов/секунду) | Rate limit, производительность |

### 8.2 Пример структуры автотестов

```
tests/
  unit/           # хендлеры с моками Bot, Dispatcher
  integration/    # с реальной БД/Redis (testcontainers или dev)
  e2e/            # Telethon: реальный бот, проверка ответов по содержимому
  load/           # скрипты locust или кастомные (отправка апдейтов)
```

### 8.3 Пример E2E с Telethon (псевдокод)

```python
# Требуется API_ID, API_HASH, сессия юзера; бот должен быть запущен
import asyncio
from telethon import TelegramClient
from telethon.tl.types import InputPeerUser

async def test_bot_start():
    client = TelegramClient(session, api_id, api_hash)
    await client.connect()
    bot = await client.get_entity("@YourBot")
    await client.send_message(bot, "/start")
    await asyncio.sleep(2)
    messages = await client.get_messages(bot, limit=1)
    assert "старт" in messages[0].text.lower() or "привет" in messages[0].text.lower()
    await client.disconnect()
```

### 8.4 Что автоматизировать в первую очередь

| Приоритет | Тип теста | Описание |
|-----------|-----------|----------|
| High | Unit: команды /start, /help, неизвестная команда | Мок Update, вызов handler, проверка вызова answer с нужным текстом |
| High | Unit: валидация пустого/длинного ввода | Передать message с text="" или 5000 символов → проверка ответа без падения |
| High | Integration: webhook оплаты (идемпотентность) | Два POST с одним payment_id → одна запись в БД |
| Medium | E2E: /start → меню → одна кнопка | Telethon: отправка /start, проверка наличия кнопки или текста меню |
| Medium | Load: 50 сообщений от одного user_id | Проверка ответа «Подождите» после N-го сообщения |
| Low | E2E: полный флоу заказа/действия | По сценарию скрипта A из плана тестирования |
