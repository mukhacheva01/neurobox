# Проверка безопасности бота (потенциал 500 активных пользователей)

**Дата:** 2026-02-16  
**Масштаб:** до 500 активных пользователей

---

## 1. Аутентификация и авторизация

| Компонент | Статус | Комментарий |
|-----------|--------|-------------|
| Telegram-бот | ✅ | Авторизация по `from_user.id`; админка по `admin_ids` / `admin_tg_ids`. |
| Веб-админка | ✅ | Логин/пароль из env, сессия с `ADMIN_PANEL_SECRET`. Рекомендуется сильный пароль и уникальный секрет на каждый инстанс. |
| Admin API (FastAPI) | ✅ | Защита через `Authorization: Bearer {ADMIN_API_SECRET_KEY}`. При пустом ключе возвращается 503. |
| Доступ к данным по user_id | ✅ | Запросы к БД используют параметризованные запросы с `user_id` из контекста Telegram. |

**Рекомендация:** Не отключать проверку IP для webhook ЮKassa в production (`SKIP_YOOKASSA_IP_CHECK` только за прокси с доверенным X-Forwarded-For).

---

## 2. Защита от злоупотреблений (rate limit)

| Механизм | Статус | Параметры |
|----------|--------|-----------|
| Rate limit на сообщения/колбэки | ✅ | 45 запросов / 60 сек на пользователя (Redis + in-memory fallback). Админы исключены. |
| Бан-лист | ✅ | Middleware проверяет `users.is_blocked`, кеш в Redis 60 сек. |
| Очередь задач (video и др.) | ✅ | Redis `neurobox:tasks`; потребление через worker. |

Для 500 пользователей: 500 × (45/60) ≈ 375 запросов/мин в пике — укладывается в типичные лимиты Telegram и backend.

---

## 3. SQL-инъекции

| Место | Статус |
|-------|--------|
| Запросы к БД | ✅ | Используется asyncpg с плейсхолдерами `$1`, `$2`. Имена колонок (`col`) берутся только из фиксированных словарей `_MODEL_COLUMNS` / `col_map`, не из пользовательского ввода. |

---

## 4. Секреты и логирование

| Элемент | Статус |
|---------|--------|
| Секреты в коде | ✅ | Токены и ключи только из `config.settings` (env). Дефолты в settings — условные (например, для локальной разработки). |
| Логи | ✅ | Structlog: процессор `_redact_sensitive` маскирует ключи, содержащие `token`, `password`, `secret`, `api_key`, `key`. |
| URL с токеном бота | ⚠️ | В коде есть `f"https://api.telegram.org/file/bot{message.bot.token}/..."` — токен не логируется, но при утечке логов URL мог бы попасть; лучше не логировать полные URL к api.telegram.org. Сейчас в логах такие URL не видны, т.к. не передаются в event_dict. |

---

## 5. Prompt injection и контент

| Элемент | Статус |
|---------|--------|
| Фильтр prompt injection | ✅ | `_is_prompt_injection()` в `bot/handlers/text.py` отсекает типичные фразы; при срабатывании запрос не отправляется в LLM, пишется warning. |
| Системный промпт | ✅ | Жёсткий системный промпт (DEFAULT_SYSTEM) и ограничение длины ввода. |

---

## 6. Webhook (ЮKassa, CryptoBot)

| Элемент | Статус |
|---------|--------|
| ЮKassa | ✅ | Проверка IP по официальному списку; при необходимости за прокси — корректный X-Forwarded-For. Идемпотентность через статус платежа в БД. |
| CryptoBot | ✅ | Обработка по `update_type` и данным счёта. |
| Секрет webhook | ⚠️ | Отдельный секретный токен в теле webhook ЮKassa не проверяется (достаточно IP). Для повышенной параноии можно добавить проверку подписи/секрета, если ЮKassa поддерживает. |

---

## 7. Ресурсы под 500 пользователей

| Ресурс | Текущее значение | Рекомендация |
|--------|-----------------|--------------|
| PostgreSQL pool (bot, worker, api) | `max_size=15` | Увеличить до **25–30** при пиковой нагрузке ~500 активных. |
| Redis (bot) | `max_connections=20` | Достаточно для rate limit и кеша; при росте воркеров можно 30. |
| Redis (память) | 512 MB, allkeys-lru | Нормально для сессий и rate limit. |
| Контейнеры | memory limits 256–512 MB | При росте нагрузки мониторить и при необходимости поднять. |

---

## 8. Веб-админка (Flask)

| Риск | Статус | Рекомендация |
|------|--------|--------------|
| Брутфорс логина | ✅ | Лимит: 5 неудачных попыток с одного IP за 15 мин. | — |
| Сессия | ✅ | Secret из env, cookie-сессия. | Использовать HTTPS в production. |
| CSRF | ⚠️ | Явной защиты нет; действия через GET (редиректы) и POST форм. | В production — HTTPS; при необходимости добавить Flask-WTF/CSRF для форм. |

---

## 9. Итоговая оценка

- **Готовность к 500 пользователям:** да, при увеличении pool БД и (по желанию) rate limit на логин админки.
- **Критичных уязвимостей:** не выявлено.
- **Улучшения:** увеличение `max_size` пула БД; ограничение попыток входа в админку по IP; в production — HTTPS, сильные пароли и уникальные секреты для каждого бота.
